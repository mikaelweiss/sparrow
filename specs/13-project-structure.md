# Project Structure

## Overview

Sparrow projects start with 2 files and grow organically. There are no mandatory directory conventions — organize your files however you want. Sparrow discovers components by type conformance, not file location.

Everything is Swift. No TOML, no YAML, no JSON config files.

## Minimal Project (sparrow new)

```
MyApp/
  App.swift            # entry point: config, routes, views
  Theme.swift          # theme customization (optional, generated with defaults)
```

This is a complete, runnable project. `sparrow run` works immediately.

## App.swift — The Entry Point

All configuration lives in Swift, in the App struct:

```swift
import Sparrow

@main
struct MyApp: App {
    var config: some Config {
        Server(port: 3000)
        Database(.postgres("$DATABASE_URL"))
        Auth(.builtin)
    }

    var body: some Scene {
        Routes {
            Page("/") {
                VStack(spacing: 16) {
                    Text("Welcome to MyApp")
                        .font(.largeTitle)
                    Text("Edit App.swift to get started.")
                        .foreground(.textSecondary)
                }
                .padding(32)
            }
        }
    }
}
```

### Config Block

The `config` property declares all infrastructure:

```swift
var config: some Config {
    // Server
    Server(port: 3000, host: "0.0.0.0")

    // Database — pick one adapter
    Database(.postgres("$DATABASE_URL"))
    Database(.postgres("$DATABASE_URL", pool: .init(maxConnections: 20)))
    Database(.convex("$CONVEX_URL"))
    Database(.supabase("$SUPABASE_URL", key: "$SUPABASE_KEY"))

    // Auth — pick one adapter
    Auth(.builtin)
    Auth(.builtin, oauth: [.google(clientId: "$GOOGLE_ID", clientSecret: "$GOOGLE_SECRET")])
    Auth(.convex)
    Auth(.supabase("$SUPABASE_URL", key: "$SUPABASE_KEY"))

    // Email — pick one adapter
    Email(.resend(apiKey: "$RESEND_KEY"))
    Email(.ses(region: "us-east-1"))
    Email(.smtp(host: "smtp.example.com", port: 587, username: "$SMTP_USER", password: "$SMTP_PASS"))

    // Analytics
    Analytics(.builtin)

    // Admin dashboard
    Admin(.enabled, path: "/_admin")
}
```

Only one of each type is allowed. The compiler enforces this.

### Environment Variables

Environment variables are referenced with `$` prefix in string literals and resolved at runtime:

```swift
Database(.postgres("$DATABASE_URL"))
```

For local development, create a `.env` file in the project root:

```
DATABASE_URL=postgres://localhost:5432/myapp
RESEND_KEY=re_abc123
```

`.env` is gitignored by default. Sparrow loads it automatically in development mode.

### Conditional Config

Because it's Swift, you can use `#if`:

```swift
var config: some Config {
    #if DEBUG
    Server(port: 3000)
    Database(.postgres("postgres://localhost:5432/myapp_dev"))
    #else
    Server(port: 8080)
    Database(.postgres("$DATABASE_URL"))
    #endif

    Auth(.builtin)
}
```

## Theme.swift

Theme customization lives in a dedicated file:

```swift
import Sparrow

extension Theme {
    static let app = Theme.default
        .primary("#6366F1")
        .secondary("#8B5CF6")
        .accent("#F59E0B")
        .cornerRadius(.md)
        .fonts(body: "Inter", heading: "Inter", mono: "JetBrains Mono")
        .dark { dark in
            dark.background("#0F172A")
                .surface("#1E293B")
        }
}
```

Used in App.swift:

```swift
@main
struct MyApp: App {
    var theme: some Theme { .app }
    // ...
}
```

Or inline if you prefer:

```swift
var theme: some Theme {
    Theme.default
        .primary("#007AFF")
}
```

The developer has full control over every design token. If `theme` is omitted, Sparrow uses `Theme.default`.

## File Discovery Rules

1. All `.swift` files in the project root and subdirectories are included in compilation
2. Exception: `.build/`, `Migrations/`, and hidden directories (`.git/`, etc.) are excluded from auto-discovery
3. `Migrations/` files are compiled separately and only run via `sparrow migrate`
4. `Assets/` contents are copied to the build output's `public/` directory as-is
5. The `@main` struct (conforming to `App`) is the entry point — there must be exactly one

## Package.swift Generation

Sparrow uses Swift Package Manager under the hood but abstracts it away. The developer never writes `Package.swift`.

`sparrow run` and `sparrow build`:
1. Read your Swift files and App config
2. Generate `Package.swift` with the Sparrow framework dependency + any extras
3. Run `swift build`
4. Start the server (for `sparrow run`)

Package.swift is a build artifact:
- Regenerated on every `sparrow run` / `sparrow build`
- Gitignored by default
- The developer never edits it

### Adding Dependencies

```swift
// In App.swift or any file
import SparrowCharts  // Sparrow resolves this from the package registry
```

Or explicitly in the config:

```swift
var config: some Config {
    Dependencies {
        .package(url: "https://github.com/sparrow-packages/charts", from: "1.0.0")
    }
}
```

CLI shortcuts:

```
sparrow add sparrow-charts      # add a dependency
sparrow remove sparrow-charts   # remove a dependency
sparrow update                  # update all dependencies
```

## Assets

Static files in `Assets/` are served at `/assets/`:

```swift
Image("logo.png")    // serves from /assets/logo.png
```

Supported asset types: images (png, jpg, gif, svg, webp), fonts (woff, woff2, ttf, otf), favicon.ico.

Assets are copied to `.build/release/public/assets/` during `sparrow build`.
