# Project Structure

## Overview

Sparrow projects start with 2 files and grow organically. There are no mandatory directory conventions — organize your files however you want. Sparrow discovers components by type conformance, not file location.

## Minimal Project (sparrow new)

```
MyApp/
  Sparrow.toml        # project configuration
  App.swift            # entry point, routes, views
```

This is a complete, runnable project. `sparrow run` works immediately.

## Small Project (a few pages)

```
MyApp/
  Sparrow.toml
  App.swift
  HomeView.swift
  AboutView.swift
  ProfileView.swift
  User.swift
```

All `.swift` files in the project directory (and subdirectories) are automatically included in the build. No need to register them anywhere.

## Sparrow.toml — Full Reference

```toml
name = "MyApp"
swift-tools-version = "6.2"

# Server
[server]
port = 3000                          # dev server port
host = "0.0.0.0"                     # bind address

# Theme
[theme]
preset = "default"                    # "default" or "minimal"
primary = "#007AFF"                   # override individual tokens
cornerRadius = "md"
fontBody = "Inter"

[theme.dark]
# override specific dark mode tokens

# Database
[database]
url = "postgres://localhost:5432/myapp"
# or
url = "$DATABASE_URL"                 # from environment

[database.pool]
maxConnections = 20                   # connection pool size

# Auth
[auth]
enabled = true
sessionDuration = "30d"
secureCookies = true

[auth.password]
minLength = 8

# Dependencies
[dependencies]
# sparrow-charts = { version = "1.0", source = "https://github.com/sparrow-packages/charts" }

# Build
[build]
# Additional swift build flags if needed
swiftFlags = []
```

## File Discovery Rules

1. All `.swift` files in the project root and subdirectories are included in compilation
2. Exception: `.build/`, `Migrations/`, and hidden directories (`.git/`, etc.) are excluded from auto-discovery
3. `Migrations/` files are compiled separately and only run via `sparrow migrate`
4. `Assets/` contents are copied to the build output's `public/` directory as-is
5. The `@main` struct (conforming to `App`) is the entry point — there must be exactly one

## Package.swift Generation

Sparrow generates `Package.swift` from `Sparrow.toml` on every build. The generated file is placed in the project root but should be treated as a build artifact:

- It's regenerated on every `sparrow run` / `sparrow build`
- It's gitignored by default (`.gitignore` includes `Package.swift` and `.build/`)
- If the developer needs to customize it beyond what `Sparrow.toml` supports, they can add a `[package]` section to the toml for additional SPM settings

## .gitignore (auto-generated)

```
.build/
Package.swift
Package.resolved
.sparrow/
```

## .sparrow/ Directory

Sparrow creates a `.sparrow/` directory for internal state:

```
.sparrow/
  generated/
    sparrow.css              # generated stylesheet
  cache/
    last_build_hash          # for incremental build detection
```

This is gitignored and ephemeral. Package.swift is always generated in the project root (SPM requires it there).

## Assets

Static files in `Assets/` are served at `/assets/`:

```swift
Image("logo.png")    // serves from /assets/logo.png
```

Supported asset types: images (png, jpg, gif, svg, webp), fonts (woff, woff2, ttf, otf), favicon.ico.

Assets are copied to `.build/release/public/assets/` during `sparrow build`.

## Environment Variables

Sparrow supports environment variables in `Sparrow.toml` via `$VAR_NAME` syntax:

```toml
[database]
url = "$DATABASE_URL"
```

For local development, create a `.env` file in the project root:

```
DATABASE_URL=postgres://localhost:5432/myapp
```

`.env` is gitignored by default. Sparrow loads it automatically in development mode.
